# KPMR Source Tree Schema Documentation

## Overview

The KPMR Source Tree XSD defines the XML structure for representing hierarchical relationships between KPMR (Kualitas Penerapan Manajemen Risiko) assessment factors. This schema is used to validate XML output generated by the `kpmr_source_tree_cli` tool.

**Schema File**: `schema/kpmr_source_tree.xsd`

## Purpose

The KPMR source tree structure represents the hierarchical organization of risk assessment factors used in Indonesian financial institutions (LJK - Lembaga Jasa Keuangan). The tree structure helps organize assessment factors into logical groupings based on their code identifiers.

## Root Element

### `<KpmrSourceTree>`

The root element that contains all top-level KPMR source nodes.

**Child Elements**:
- `<KpmrSourceNode>` (0 or more) - Top-level assessment factor nodes

## Node Structure

### `<KpmrSourceNode>`

Represents a single assessment factor node in the tree. Nodes can be nested to represent parent-child relationships.

**Child Elements**:

| Element | Type | Required | Description |
|---------|------|----------|-------------|
| `code` | string | Yes | Unique identifier code (e.g., SK01001000, SK01001001) |
| `risk_id` | string | Yes | Risk type identifier |
| `assesment_factor` | string | Yes | Description of the assessment factor (in Indonesian) |
| `pic` | string | Yes | Person In Charge information in JSON format |
| `KpmrSourceNode` | recursive | No | Child nodes (0 or more) |

## Code Structure and Hierarchy

### Code Format

Codes follow the pattern: `SKxxxxxxxxx` where:
- `SK` = Prefix for KPMR codes
- Followed by numeric segments

### Parent-Child Relationships

The hierarchical structure is determined by the code pattern:

**Parent Nodes**: Codes ending with `000`
```
SK01001000 - Tata Kelola (Risk Governance)
SK01002000 - Kerangka Manajemen Risiko
SK01003000 - Proses Manajemen Risiko, SIM dan SDM
SK01004000 - Sistem Pengendalian
```

**Child Nodes**: Share the parent's prefix but have different endings
```
SK01001000 (parent)
  ├── SK01001001 (child)
  ├── SK01001002 (child)
  ├── SK01001003 (child)
  └── SK01001004 (child)
```

## Field Descriptions

### code
The unique identifier for each assessment factor. Format follows the pattern where parent nodes end with `000` and child nodes share the prefix with different numeric endings.

**Examples**:
- `SK01001000` - Parent node
- `SK01001001` - First child of SK01001000
- `SK01001002` - Second child of SK01001000

### risk_id
Identifier for the type of risk being assessed. Common value is `8` for credit risk (Risiko kredit).

**Example**: `8`

### assesment_factor
Detailed description in Indonesian of what the assessment factor evaluates. This describes the specific aspect of risk management being assessed.

**Examples**:
- "Tata Kelola (Risk Governance)"
- "Penetapan tingkat Risiko kredit yang akan diambil oleh LJK dan toleransi Risikonya sejalan dengan sasaran strategis dan strategi bisnis LJK secara keseluruhan"

### pic
Person In Charge information stored as a JSON string. Specifies which entities or individuals are responsible for this assessment factor.

**JSON Structure**:
```json
{
  "value": "string",
  "from_col": "string"
}
```

**Common Patterns**:

1. **All entities** (wildcard):
   ```json
   {"value": "*", "from_col": "ALL"}
   ```

2. **Specific company types**:
   ```json
   {"value": "BANK,INSURANCE_COMPANY", "from_col": "COMPANY_TYPE"}
   ```

3. **Individual entities**:
   ```json
   {"value": "1,100", "from_col": "INDIVIDUAL"}
   ```

## XML Example

```xml
<?xml version="1.0" encoding="UTF-8"?>
<KpmrSourceTree>
  <KpmrSourceNode>
    <code>SK01001000</code>
    <risk_id>8</risk_id>
    <assesment_factor>Tata Kelola (Risk Governance)</assesment_factor>
    <pic>{"value": "*", "from_col": "ALL"}</pic>
    <KpmrSourceNode>
      <code>SK01001001</code>
      <risk_id>8</risk_id>
      <assesment_factor>Penetapan tingkat Risiko kredit yang akan diambil oleh LJK dan toleransi Risikonya sejalan dengan sasaran strategis dan strategi bisnis LJK secara keseluruhan</assesment_factor>
      <pic>{"value": "*", "from_col": "ALL"}</pic>
    </KpmrSourceNode>
    <KpmrSourceNode>
      <code>SK01001002</code>
      <risk_id>8</risk_id>
      <assesment_factor>Awareness dan pemahaman Dewan Komisaris dan Direksi LJK mengenai manajemen Risiko kredit</assesment_factor>
      <pic>{"value": "BANK,INSURANCE_COMPANY", "from_col": "COMPANY_TYPE"}</pic>
    </KpmrSourceNode>
  </KpmrSourceNode>
</KpmrSourceTree>
```

## Validation

To validate an XML file against this schema, use:

```bash
xmllint --noout --schema schema/kpmr_source_tree.xsd your_file.xml
```

**Success Output**:
```
your_file.xml validates
```

## Usage with CLI Tool

The `kpmr_source_tree_cli` tool generates XML conforming to this schema:

```bash
# Generate XML output
./kpmr_source_tree_cli -i data/kpmr_source.xml -f xml > output.xml

# Validate the output
xmllint --noout --schema schema/kpmr_source_tree.xsd output.xml
```

## Schema Compliance

This schema enforces:
- ✅ Proper nesting of nodes
- ✅ Required fields (code, risk_id, assesment_factor, pic)
- ✅ Recursive structure for parent-child relationships
- ✅ Well-formed XML structure

## Notes

1. The schema allows for unlimited nesting depth through the recursive `KpmrSourceNode` type
2. All fields are currently defined as strings for maximum flexibility
3. The PIC field contains JSON data but is stored as a string in XML
4. The hierarchical relationship is semantic (based on code patterns) rather than explicitly enforced by the schema
